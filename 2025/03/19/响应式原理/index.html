什么是数据劫持？

就是通过Object.defineProperty的getter和setter控制访问和修改

相当于我们自定义了obj.a取值和赋值的行为，使用自定义的getter和setter来重写了原有的行为，就是数据劫持的含义

为了保证数据的私有性，我们将上述过程变为一个闭包

Observer类与observe函数

那如果obj有多个属性呢？我们就建立一个Observer类来遍历对象
Observe类的作用：

  1. 先给每个Observe实例都添加一个__ob__属性(相当于一个标记，标记着该属性已经被数据劫持)
  2. 遍历该层的每一个属性，利用defineReactive,将他们全都变成响应式的数据
  3. dep处理数组变异的方法 其中def函数的作用就是设置传入属性的配置

observe函数

作用就是判断当前传入参数是不是一个响应式的值

先用observe观测该属性是否有__ob__属性(有__ob__就意味着已经是Observe类),如果有，就不做处理，如果没有，就new 一个Observer类；该函数返回的是__ob__属性。
想要继续往下遍历嵌套的属性，那么就需要在目前最后一步，也就是defineReactive中再次观测嵌套的属性

关于数组

observe不做改变，Observer中多增加一个判断(判断是数组还是对象)；如果是对象，还是用walk遍历属性；但如果是数组，先把它的原型对象改为arrayMethods，然后再定义一个observeArray方法用来遍历数组中的每一个元素。
但这里可能会有疑问：为什么不能像普通对象那样直接defineReactive呢？

  1. 如果为每个元素都添加一个defineReactive，数组长度可能极大（如数万元素），若为每个索引都定义 getter/setter，初始化性能极差。数组的 push、pop、splice 等方法会动态修改索引，导致需要不断劫持新索引，性能不可控。
  2. 原生方法无法拦截：直接通过索引修改数组（如 arr[0] = 1）可以被 defineReactive 捕获，但类似 push、splice 等原生方法修改数组时，无法通过索引劫持感知变化。

数组的处理逻辑：
所有的数组都会被添加__ob__属性；(数组本身都会被observe函数处理，并添加__ob__属性)
Vue 会递归观测数组中的元素，如果元素是对象或者数组，就会添加__ob__属性，如果是基本数据类型，就不会被添加。

对于数组的’push’,’pop’,’shift’,’unshift’，’splice’,’sort’,’reverse’这几个方法，不会触发属性的setter,所以我们要对他们进行改写

收集依赖与派发更新

举例：演唱会抢票时，我们可以点击预售提醒，当开票时，大麦就会提醒我们抢票，我们看到消息后就可以开始购买。点击预约抢票，就相当于在大麦上登录了自己的信息(订阅),大麦就会把我们的信息存储起来。开票的时候，大麦会通知所有订阅者，我们就会去抢票(进行一系列动作)

在这个过程中，我们就相当于Watcher。每个Watcher实例订阅一个或多个数据，这些数据被称为Watcher的依赖(商品就是买家的依赖)；当依赖发生变化，Watcher实例会接收到数据发生变化这条消息，之后会执行一个回调函数来实现某些功能，比如更新页面(买家进行一些动作)。

过程：

  1. 初始化：
    当 Vue 实例创建时，会对每个需要响应的数据属性(先observe)创建一个 Watcher 实例。这些 Watcher 可能是与视图相关的（即渲染函数），也可能是与计算属性相关的。
  2. 依赖收集阶段：
    在执行渲染函数或计算属性的过程中，如果访问了某个响应式数据属性，Dep 会记录下当前的 Watcher 作为该属性的依赖。
  3. 数据变更检测：
    当数据发生变化时，Dep 会通知所有依赖于该数据的 Watcher 进行更新。
  4. 触发更新：
    Watcher 收到通知后，会重新执行其关联的回调函数（如渲染函数或计算属性的重新计算），从而实现视图的自动更新或计算属性值的重新计算。

Dep类主要用于依赖收集和通知更新。

一共有两个地方添加了Dep类

  1. defineReactive中：为每个响应式属性创建一个独立的Dep实例，用于管理对该属性的依赖关系
  2. Observer 类中的 Dep：为整个对象创建一个全局的 Dep 实例，主要用于处理数组的变异方法如 push, pop, shift, unshift, splice, sort, reverse）。这些方法不会触发单个属性的 getter 和 setter，因此需要一个全局的 Dep 来管理整个对象的依赖关系。

数组响应式的总结

在组件初始化时，Vue会先遍历data选项中的对象，调用observe将其转换为响应式。此时，数据已经具备响应式能力，但还没有进行依赖收集。当组件开始渲染时，会创建一个渲染Watcher，这个Watcher在执行渲染函数时会访问数据属性，触发getter，从而将Watcher添加到对应的dep中。这时候依赖收集才真正发生。
依赖收集实际上是在访问这些属性的时候触发的。
例：当在组件的渲染过程中访问了某个属性，这时候getter被触发，才会将当前的Watcher（比如渲染Watcher）添加到这个属性的依赖列表中

流程解析：

  1. 初始化阶段：observe(obj) 为 obj 的 child 属性定义 getter/setter。observe(obj.child) 递归观测子对象，为 x 定义 getter/setter。
  2. 依赖收集阶段：Watcher 执行计算函数，访问 obj.child.x：触发 obj.child 的 getter → 收集 obj.child 的 dep。触发 obj.child.x 的 getter → 收集 x 的 dep。同时收集 childOb.dep（子对象的依赖管理器）。
  3. 触发更新：修改 obj.child.x = 2 → 触发 x 的 setter → 通知 x 的 dep。由于 childOb.dep 也收集了 Watcher，子对象的更新会通知父级 Watcher。

一些小疑问

Object.defineProperty：主要用于拦截对象属性的访问和修改。它无法直接处理数组下标的改变（如 arr[0] = ‘new value’）或数组长度的变化（如 arr.length = 5），也无法自动拦截数组方法（如 push, pop, splice 等）的调用。
Watcher：用于监听数据的变化并执行相应的回调函数。它依赖于 Dep 类来管理依赖关系，并在数据发生变化时触发更新逻辑。

为什么数组下标修改或原生方法无法触发setter？

  1. 性能问题：数组可能包含大量元素，若为每个索引都设置 getter/setter，会导致初始化性能损耗。

  2. 实现限制：Object.defineProperty 的设计目标是对象属性，而非动态长度的数组。直接通过下标修改（如 arr[0] = 1）本质上是操作对象的属性，但Vue未对数组索引劫持，因此无法触发响应式更新。

  3. 原生方法不可控：如 push/pop 等方法会修改数组长度或内容，但这些方法是 Array.prototype 的原生实现，未被Vue劫持，因此默认不会触发更新。

为什么observe函数要返回对象的__ob__属性？
__ob__ 是 Vue 在对象上挂载的一个特殊属性，指向一个 Observer 类的实例。
用途：

  1. 标记该对象已被观测(避免重复观测)
  2. 提供对嵌套对象/数组的响应式支持递归观测：当对象有嵌套属性（如 obj.a.b）时，Observer 实例会递归调用 observe，确保所有层级的属性都被劫持。数组响应式：数组的 ob 会被重写的方法（如 push、splice）用来触发依赖更新。
  3. 管理该对象的依赖(通过__ob__.dep，dep是Observe类的属性)

显式操作：
通过Vue封装的方法（如 Vue.set、数组变异方法）修改数据，这些操作主动触发依赖追踪。
隐式操作：
直接通过JavaScript原生方式操作数据（如 arr[0] = 1 或 arr.push()），这些操作绕过Vue的响应式系统，不会自动触发更新。
