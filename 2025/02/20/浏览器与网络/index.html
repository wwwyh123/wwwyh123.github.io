浏览器与网络

从输入URL到页面渲染的完整过程是什么？

  1. DNS解析：浏览器解析 URL 中的域名，查询其对应的 IP 地址。
  2. TCP连接：浏览器与服务器建立 TCP 连接（通常使用三次握手）。
  3. 发送 HTTP 请求：浏览器向服务器发送 HTTP 请求。
  4. 服务器处理请求并响应：服务器根据请求返回 HTML、CSS、JS 等资源。
  5. 浏览器渲染页面：浏览器开始解析 HTML 内容，构建 DOM 树，解析 CSS，生成 CSSOM 树，然后合并生成渲染树，最后绘制页面。

浏览器渲染页面的全过程

请求页面->请求HTML->构建DOM->请求css,构建CSSOM->构建渲染树->如果构建DOM时遇到了js那么就请求下载执行JS->布局(获取渲染树的结构，节点以及位置和大小,布局是依据盒子模型进行的，这些盒子在页面上进行排列和嵌套)->绘制(把DOM树以像素的形式绘制在页面上)

什么是跨域？CORS的原理是什么？如何解决跨域问题？

指在浏览器中，不同源的请求（不同的域名、端口或协议）会受到浏览器的安全限制。比如，前端向不同域的 API 发送请求会被浏览器拦截。
CORS（跨域资源共享）：是一种浏览器机制，允许服务器明确指定哪些源可以访问资源。通过HTTP头信息（如 Access-Control-Allow-Origin）告知浏览器，哪些请求可以跨域访问。
解决方法：

  • CORS：通过在服务器端配置响应头，允许跨域请求。
  • JSONP：利用 <script> 标签不受同源策略限制的特性来跨域。
  • 代理服务器：使用服务器代理请求，避免浏览器的跨域限制。

浏览器事件循环中宏任务（MacroTask）和微任务（MicroTask）的执行顺序是什么？

执行栈执行完毕，会去任务队列看是否有异步任务，有就送到执行栈执行，反复循环查看执行，这个过程就是事件循环

  1. 执行同步代码
  2. 执行所有微任务队列中的任务（如：Promise.then）。
  3. 再执行下一个宏任务。

HTTP缓存策略（强缓存和协商缓存）是如何工作的？

  • 强缓存：指浏览器直接使用缓存中的资源，而不向服务器发送请求。常通过 Cache-Control 或 Expires 头来控制。
    Cache-Control：是一个通用的 HTTP 头字段，用于控制缓存的行为。它可以出现在请求头和响应头中，并且可以包含多个指令:
  • public：
    表示响应可以被任何缓存存储，包括中间代理服务器。
  • private：
    表示响应只能被用户的浏览器缓存，不能被中间代理服务器缓存。
  • no-cache：
    表示在使用缓存之前必须先验证资源的有效性（即需要与服务器进行一次条件请求）。
  • no-store：
    表示禁止缓存任何内容，每次请求都必须从服务器获取最新资源。
    max-age，定义资源在客户端缓存的时长。
    **Expires**：是一个较早的 HTTP 响应头字段，用于指定资源的过期时间。它是一个绝对的时间戳，格式为 GMT 时间。
  • 协商缓存：浏览器向服务器发送请求，但服务器通过一些机制（如 Last-Modified、ETag）来判断资源是否更新，若未更新则使用缓存。
    ETag是一个唯一标识符，通常是由服务器生成的哈希值，用于标识某个特定版本的资源。当资源发生变化时，ETag 也会随之改变。(首次请求，客户端向服务器请求资源，服务器返回资源的同时，在响应头上包含ETag字段；客户端再次请求相同的资源时，会在请求头上添加If-None-Math字段，并附上之前收到的Etag值；如果资源未发生变化，即Etag未改变，服务器返回304 Not Modified 状态码，表示客户端可以使用本地缓存的资源;如果资源发生了变化，服务器返回新的资源及更新以后的ETag)
    Last-Modified 是一个时间戳，表示资源最后一次修改的时间。它通常由服务器在首次响应时提供，并且可以在后续请求中用于验证资源是否已经更新。
    首次请求：
    客户端向服务器请求资源。
    服务器返回资源的同时，在响应头中包含 Last-Modified 字段;客户端再次请求相同的资源时，会在请求头中添加 If-Modified-Since 字段，并附上之前收到的 Last-Modified 时间，服务器接收到请求后，会根据 If-Modified-Since 中的时间检查资源是否在此之后发生了变化：
    如果资源未发生变化（即自上次修改以来没有更新），服务器返回 304 Not Modified 状态码，表示客户端可以使用本地缓存的资源。
    如果资源发生了变化（即自上次修改以来有更新），服务器返回新的资源以及更新后的 Last-Modified 时间。
    以下是缓存机制的优先级顺序：
  1. Cache-Control：最高优先级，用于控制强缓存策略。如果设置了 no-store 或 no-cache，将直接跳过缓存或强制重新验证。
  2. Expires：次优先级，用于指定资源的过期时间。但在 HTTP/1.1 中，Cache-Control 的优先级更高。
  3. ETag：用于协商缓存验证，优先于 Last-Modified。
  4. Last-Modified：用于协商缓存验证，但优先级低于 ETag。

什么是XSS攻击？如何防范？

XSS（跨站脚本攻击）：攻击者将恶意脚本注入到网页中，这些脚本会在用户浏览网页时执行，导致敏感数据泄露、账号被盗等安全问题。
防范措施：
输入验证：对用户输入进行严格的验证，避免注入恶意脚本。
输出编码：对动态生成的内容进行 HTML 编码，防止脚本被执行。
HTTPOnly 和 Secure Cookie：使用 HTTPOnly 和 Secure 属性来增强 Cookies 的安全性。
CSP（内容安全策略）：通过设置 Content-Security-Policy 头来限制页面加载的资源，防止加载外部恶意脚本。

列举一些HTTP状态码

  • 1xx 服务器收到请求，需要请求者继续执行操作
  • 200 OK:请求成功，服务器返回所请求的数据
  • 3xx (重定向状态码)：需要后续操作才能完成这一请求
  • 403 Forbidden:服务器已经理解请求，但是拒接执行
  • 404 NOT FOUND:请求的资源在服务器上不存在
  • 500 Internal Server Error:服务器内部错误，无法完成请求
  • 304 Not Modified 表示客户端的缓存版本仍然是最新的，不需要重新下载资源。

浏览器渲染流程/网页加载过程（从输入URL到看到页面）

  1. URL解析：浏览器解析用户输入的 URL，提取协议、域名、端口、路径等信息

  2. DNS查询:通过 DNS 查询将域名解析为对应的 IP 地址。

  3. 建立TCP连接:浏览器与服务器之间通过三次握手建立 TCP 连接。

  4. 发送HTTP请求

  5. 服务器处理请求

  6. 接收HTTP响应

  7. 浏览器解析和渲染

  8. 加载静态资源

  9. 页面交互

图片懒加载（Lazy Loading）

一种优化网页性能的技术，它通过延迟加载页面上的非关键资源（如图片、视频等），直到用户需要查看这些资源时才进行加载。

  1. 加快页面初始加载速度：懒加载减少了页面初次加载时需要下载的数据量，使得页面能够更快地显示出来。
  2. 节省流量和带宽：对于使用移动设备或受限于网络条件的用户而言，懒加载可以显著减少不必要的数据传输。只有当用户滚动到相关内容区域时，才会加载相应的图片，从而节省了用户的流量和带宽消耗。
  3. 改善页面响应性：由于初始加载的内容较少，浏览器需要处理的任务也相应减少，这有助于提高页面的整体响应性和流畅度，尤其是在低端设备上效果更为明显。
  4. 增强用户体验：通过逐步加载内容，用户可以看到页面从无到有的构建过程，而不是长时间等待整个页面完全加载。此外，懒加载还可以结合占位符或者渐进式图像来进一步提升视觉体验，使用户在等待图片加载过程中不会感到突兀。
  5. 优先展示关键内容：懒加载允许开发者优先加载和渲染页面中最重要的部分，确保用户首先看到的是最相关的信息，而将次级内容的加载推迟到必要时刻。

预加载和懒加载

预加载：在用户请求之前提前加载某些资源，以便这些资源在需要时可以立即使用，从而减少等待时间。预加载适用于那些你确定用户即将需要但尚未请求的资源。
适用场景：关键资源、导航预测
实现方法：

  1. 通过 <link> 标签进行预加载

    rel="preload"：告诉浏览器这是一个预加载资源。
    as 属性用于指定预加载资源的类型，这对于浏览器正确处理资源优先级、缓存以及请求头信息非常重要。script 和 style 通常会被赋予较高的优先级

  2. 通过 JavaScript 动态预加载

    onload 事件用于检测资源是否已经成功加载。当预加载的资源完全加载完毕后，onload 回调函数将被触发。

懒加载：一种延迟加载技术，只有当资源进入视口（即用户可见区域）或即将进入视口时才加载资源。这有助于减少初始页面加载时间，节省带宽，并提高页面性能。
适用场景：图片、视频、非关键脚本
原理：

  1. 延迟加载： 核心思想是推迟加载那些当前不在视口（即用户可见区域）内的资源，直到这些资源即将进入视口或用户滚动到它们的位置时再进行加载。
  2. 监听事件：通过监听用户的滚动、缩放等事件，判断资源是否即将进入视口，并在此时触发加载操作。
  3. 动态插入资源：一旦确定某个资源需要加载，通过动态创建 <img> 标签、<script> 标签或其他方式将资源插入 DOM中并开始加载。

常见的请求头都有哪些?

get 和 post 的区别

  • GET 方法
    用途：主要用于请求数据，即从服务器检索信息。
  1. 参数传递：通过URL发送参数（例如 example.com/?key1=value1&key2=value2），这些参数直接显示在浏览器地址栏。
  2. 数据长度限制：由于参数附在URL之后，受限于浏览器和服务器对URL长度的限制，通常不适合传输大量数据。
  3. 缓存：可以被浏览器缓存，也可能被搜索引擎索引。
  4. 安全性：因为数据直接暴露在URL中，所以不适合用来发送敏感信息。
  • POST 方法
    用途：主要用于提交数据给服务器处理，比如向数据库添加或更新记录。
  1. 参数传递：通过HTTP请求体发送数据，不在URL中显示。
  2. 数据长度限制：理论上没有严格的数据量限制，但实际上会受到服务器配置的影响。
  3. 缓存：不会被浏览器缓存，默认情况下也不会被搜索引擎索引。
  4. 安全性：相比GET方法，POST方法更加安全，因为数据不是明文显示在URL上，但仍然需要额外的安全措施（如HTTPS）来保护数据传输安全。
  • PUT 方法
    用途：主要用于向服务器上传数据以创建或更新资源。
    特点：
  • 如果目标资源已经存在，则使用提供的数据更新该资源。
  • 如果目标资源不存在，服务器可能会根据请求创建一个新的资源，并且URL由客户端指定。
    示例场景：
    更新用户资料：如果有一个API端点 /users/123，你可以发送一个 PUT 请求来更新用户ID为123的用户信息。
    文件上传：将特定文件的内容上传到服务器上的特定位置。
  • DELETE 方法
    用途：用于请求服务器删除指定的资源。
    特点：
    发送 DELETE 请求给一个特定的资源URL，指示服务器应移除对应标识的资源。
    成功执行后，服务器可能返回状态码如 200 OK（包含被删除资源的信息）或 204 No Content（没有返回内容）。
    示例场景：
    删除用户的个人资料：发送 DELETE 请求到 /users/123 来删除用户ID为123的用户资料。
    移除文件：通过发送 DELETE 请求到相应的文件路径来从服务器上删除文件。

HTTP和HTTPS的区别

是用于在网络上传输网页的两种协议，它们之间的主要区别在于安全性：

  1. 安全性：
    HTTP：是以明文形式发送数据的，不提供任何加密或安全保障。这意味着在客户端与服务器之间传输的数据（如用户名、密码等敏感信息）可以被轻易地拦截和读取。
    HTTPS：通过SSL（Secure Sockets Layer）或TLS（Transport Layer Security）协议为通信加密，确保数据在客户端和服务器之间的传输过程中是加密的，防止中间人攻击和窃听。
  2. URL前缀：
    HTTP网址以http://开头。
    HTTPS网址以https://开头，并且很多浏览器会在地址栏显示一个锁形图标，表示连接是安全的。
  3. 端口号：
    默认情况下，HTTP使用80端口。
    HTTPS默认使用443端口。
  4. 性能影响：
    由于HTTPS需要额外的计算资源来进行加密和解密操作，理论上它可能会比HTTP稍慢一些。然而，现代硬件和优化技术（如HTTP/2通常仅通过HTTPS提供）已经大大缩小了这种差距，有时甚至让HTTPS表现得更好。
  5. SEO优势：
    Google和其他搜索引擎倾向于给予HTTPS网站更好的排名。因此，从SEO的角度来看，采用HTTPS是有益的。
    信任度：
    使用HTTPS的网站通常会获得更高的用户信任度，因为用户知道他们的数据得到了保护。此外，某些浏览器会标记HTTP站点为“不安全”，这可能会影响用户的访问意愿。

TCP,IP,HTTP是什么？

TCP (Transmission Control Protocol)

TCP 是一种面向连接的、可靠的传输层协议，用于在网络中提供可靠的数据传输服务。它确保数据包按顺序到达并且没有丢失或损坏。

IP(Internet Protocol)

IP 是网络层协议，负责将数据包从源地址传输到目标地址。它不保证数据包的可靠性和顺序性，只负责尽力而为地传输数据。

HTTP (Hypertext Transfer Protocol)

HTTP 是应用层协议，用于在Web 浏览器和 Web 服务器之间传输超文本（如 HTML 页面）。它是万维网的基础协议之一。

HTTPS是如何保证安全的？

HTTP在通信过程中，存在以下问题：

  • 通信使用明文（不加密），内容可能被窃听
  • 不验证通信方的身份，因此有可能遭遇伪装

而HTTPS的出现正是解决这些问题，HTTPS是建立在SSL之上，其安全性由SSL来保证。在采用SSL后，HTTP就拥有了HTTPS的加密、证书和完整性保护这些功能

SSL的实现这些功能主要依赖于三种手段：

  1. 对称加密：加密和解密使用的秘钥都是同一个，是对称的
  2. 非对称加密：存在两个秘钥，一个叫公钥，一个叫私钥。公钥加密后只能用私钥解密;私钥加密后也只能用公钥解密;(自己的密钥＋对方的密钥+公钥)
  3. 摘要算法：验证信息的完整性
  4. 数字签名：身份验证

DNS

DNS（Domain Names System），域名系统，是互联网一项服务，把域名转化为相应的IP地址。

CDN 目的是为了改善互联网的服务质量，通俗一点说其实就是提高访问速度

CDN 构建了全国、全球级别的专网，让用户就近访问专网里的边缘节点，降低了传输延迟，实现了网站加速
