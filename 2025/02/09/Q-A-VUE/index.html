MVVM

MVVM 架构主要由三个核心部分组成：

  1. Model对应data中的数据
  2. View（视图）看到的页面
  3. ViewModel（视图模型）Vue实例
    核心特性：
    MVVM 的一个关键特性是双向数据绑定。这意味着当 Model 中的数据发生变化时，View 会自动更新；同样，当用户在 View 中修改数据时，Model 也会自动更新。这种机制大大减少了手动同步数据的工作量。
    视图模型是MVVM模式的核心，它有两个方向：一是将模型转化成视图(后端的数据转化成所看到的页面)实现方式是数据绑定。二是将视图转化成模型，即将所看到的页面转化成后端的数据。实现的方式是DOM事件监听。这两个方向都实现，我们称之为数据的双向绑定。

vue3生命周期

Vue实例从创建到销毁的过程，就是生命周期
beforeCreate：实例初始化之后，数据观测(data observer) 和 event/watcher 事件配置之前。
created：实例创建完成后被调用。此时已完成数据观测(data observer)，属性和方法的运算，watch/event 事件回调。但尚未挂载(mounted)，$el 属性不可见。
beforeMount：在挂载开始之前被调用：相关的 render 函数首次被调用。
mounted：实例挂载到DOM后调用，这时可以访问到 $el 属性。
beforeUpdate：数据更新时，在虚拟 DOM 打补丁之前调用。适合在更新前访问现有的 DOM。
updated：由于数据更改导致的虚拟 DOM 重新渲染和打补丁，在这之后会调用该钩子。
beforeUnmount（在 Vue 2 中名为 beforeDestroy）：实例销毁之前调用。在这一步，实例仍然完全可用。
unmounted（在 Vue 2 中名为 destroyed）：实例销毁后调用。调用后，所有的事件监听器会被移除，所有子实例也会被销毁。

Vue的生命周期钩子中，created和mounted的区别是什么？

  • created：实例已创建，但DOM未挂载（不可操作DOM）。
  • mounted：DOM已挂载到页面，可操作DOM。
    适用场景：
  • created：初始化数据（如API请求）。
  • mounted：操作DOM或集成第三方库（如初始化图表）。

假设现在有一个父组件，加两个子组件，说一下他们的生命周期实现顺序。

  1. 父组件的beforeCreate和created钩子执行。
  2. 父组件的beforeMount钩子执行。
  3. 子组件1的beforeCreate、created、beforeMount和mounted依次执行。
  4. 子组件2的beforeCreate、created、beforeMount和mounted依次执行。
  5. 父组件的mounted钩子最后执行。

这样，父组件在挂载前会先确保所有子组件都已创建并挂载完成，保证在父组件的mounted钩子中可以访问到子组件的DOM。

在更新阶段，父组件的beforeUpdate会先触发，然后子组件的beforeUpdate和updated依次执行，最后父组件的updated触发。销毁阶段则是父组件的beforeDestroy先执行，然后子组件的beforeDestroy和destroyed依次执行，最后父组件的destroyed触发。

需要解释Vue的渲染顺序：父组件需要先解析模板，生成子组件的实例，并完成子组件的挂载后，才能完成自身的挂载。这样才能确保父组件挂载时，子组件已经就绪。

父子组件生命周期顺序：父组件创建 → 子组件创建和挂载 → 父组件挂载。

核心规则：父组件的挂载（mounted）一定在所有子组件挂载完成后触发。

应用场景：在父组件的 mounted 中操作子组件的 DOM或数据是安全的。

vue3 和 vue2的区别

  1. 响应式系统
  • vue2
    • 使用Object.defineProperty 实现响应式。
    • 无法检测对象属性的新增或删除(需要用Vue.set/Vue.delete)
    • 对数组的某些方法()
  • Vue3
    • 改用Proxy实现响应式
    • 支持对象和数组的动态属性增删
  1. 组合式API
    • Vue2
      • 基于Options API（data、methods、computed 等选项）。
      • 逻辑分散，大型组件代码组织困难。
    • Vue3
      • 引入 Composition API（setup 函数 + 响应式函数）。
      • 逻辑可复用和组合，代码更集中、灵活。
  2. 生命周期与钩子
    • Vue2
      • 生命周期钩子有beforeCreate,created,beforeMount,mounted,beforeUpdate,updated,beforeDestroy,destroyed
      • 在beforeCreate与created执行后，问你有没有vm.$mount(el)，如果没有，就走不下去
    • Vue3
      • 钩子更名destroyed → unmounted，beforeDestroy → beforeUnmount
      • 完全准备好所有再执行(app=Vue.createApp(opyions) app.mount(el))
  3. 指令与插槽
    • vue2中使用slot可以直接使用slot,而vue3中必须使用v-slot的形式
    • v-for与v-if在vue2中优先级高的是v-for指令，而且不建议一起使用
    • vue3中v-for与v-if,只会把当前v-if当做v-for中的一个判断语句，不会相互冲突
    • vue3中移除keyCode作为v-on的修饰符，当然也不支持config.keyCodes
    • vue3中移除v-on.native修饰符
    • vue3中移除过滤器filter
  4. ref和reactive的区别
    • ref定义的是基本数据类型
    • ref通过Object.defineProperty()的get和set实现数据劫持
    • ref操作数据.value，读取时不需要value
    • reactive定义对象或数组数据类型
    • reactive通过Proxy实现数据劫持
    • reactive操作和读取数据不需要.value
  5. 性能提升：
    更快的渲染速度：Vue 3 采用了虚拟 DOM 的重写，并引入了静态树提升、静态属性提升等优化技术，使得渲染速度得到了显著提升。
    更小的包体积：通过Tree-shaking的支持（移除未使用的代码），Vue 3 的核心库可以做到更小，只有当实际使用到的功能才会被打包进去，从而减少了最终打包文件的大小。

什么是虚拟DOM？为什么需要它？

虚拟DOM（Virtual DOM） 是一个轻量级的JavaScript对象，描述真实DOM的结构。
作用：
性能优化：通过Diff算法对比新旧虚拟DOM，最小化真实DOM操作。
跨平台：抽象渲染层（如渲染到Web、Native）。
Vue中的虚拟DOM：通过render函数生成，更新时对比差异后局部更新真实DOM。

数据代理

通过vm对象来代理data对象中的属性的操作。vm身上有的是_data，而没有data。然后通过Object.definePropoty的getter读取到_data里的数据复制一份到vm身上。
好处：更加方便的操作data中的数据
基本原理：通过Object.defineProperty()把data对象中的所有属性添加到vm上。为每一个添加到vm上的属性都指定一个getter和setter。在getter/setter内部去操作（读/写）data中对应的属性。

数据双向绑定的基本原理

采用数据劫持结合发布者-订阅者模式的方式。
数据劫持：通过Object.defineProperty()来劫持对象各个属性的getter,setter
发布订阅者模式：在数据变动时发布者发布消息给订阅者，触发相应监听回调
当把一个普通的js对象传给vue实例来作为它的data选项时，vue将遍历它的属性，用Object.defineProperty()监听它们的getter/setter方法。这样，他们就可以让vue追踪依赖，在对象的属性被访问和修改时通知变化

Vue2和Vue3在响应式原理上的主要区别是什么？

Vue2:使用Object.defineProperty()监听数据变化，访问时触发getter，修改时触发setter来更新视图，但它有局限性：无法监听数组索引的内容更改和新属性的添加与删除。需要借助(Vue.set())
Vue3:采用了 Proxy 来实现响应式，它能劫持整个对象及其嵌套的对象，能够监听数组索引、对象的新增属性等，性能和灵活性更好。

v-for为什么设置key

  1. 帮vue识别元素：keys是一个特殊的属性，用于帮助vue识别每一个节点的身份，以便在更新DOM时更高效的进行差异对比，从而有效地更新和重排序元素；没有 key，Vue 只能通过元素在数组中的索引进行追踪，这在某些情况下（如动态列表）可能导致不必要的重渲染或错误的状态管理。
  2. 提升渲染效率
    当列表数据发生变化（如添加、删除或重新排序），Vue 需要决定如何高效地更新DOM。拥有唯一key 的元素可以让Vue快速定位到具体哪些元素发生了变化，而不是简单地认为所有元素都需要重新渲染。这样可以显著提升应用的性能。
  3. 维持组件状态
    如果列表项是复杂的子组件，且这些组件有自己的内部状态，那么不设置 key 或者使用索引作为 key 可能会导致子组件状态丢失。这是因为当列表顺序改变时，如果没有唯一标识符，Vue可能会销毁并重建组件而非简单地移动它们。通过设置稳定的 key，可以确保组件状态得到正确的保留。

Vue中的nextTick是什么？它的作用是什么？原理？

Vue 提供的一个 API，用于在 DOM 更新后执行某些操作。Vue 会将 DOM 更新操作和 JavaScript 执行操作放入异步队列中。
使用场景： 1. 修改数据后立即操作DOM 2. 确保视图更新后再执行某些逻辑
nextTick利用 JavaScript 的事件循环机制，优先使用微任务来执行回调如 Promise.then()、MutationObserver 或者 setImmediate（对于 Node.js 环境），确保在 DOM 更新后触发

Vue的v-if和v-show有什么区别？性能上有何差异？

v-if：根据条件动态地创建和销毁 DOM 元素。当条件为 false 时，元素会被销毁；当条件为 true 时，元素会被创建。适用于条件变化不频繁的场景。v-if有更高的切换消耗
v-show：通过切换元素的 display 样式来控制元素显示与否，元素会始终存在 DOM 中，只是改变其 CSS 样式的显示或隐藏。适用于频繁切换的场景。v-show有更高的初始渲染消耗；

什么是Vue的混入（Mixin）？它的优缺点是什么？

Mixin 是 Vue 提供的一种机制，它允许将多个组件的逻辑提取并共享到不同的组件中。通过定义一个包含 data、methods、computed，生命周期钩子 等的混入对象，在多个组件中复用该逻辑。
优点：代码复用性高，能够避免逻辑的重复。
缺点：可能导致命名冲突（尤其是当多个 mixin 中有相同的方法、数据时），会增加调试和维护的复杂度。大规模使用时，可能让代码变得难以追踪。

Vue3的<script setup>语法糖解决了什么问题？

<script setup>是 Vue3 提供的一种新语法糖，用于简化组件的编写。它通过<script setup>标签，减少了重复的 import 和 return，并且优化了组件的编译过程，提升了性能。它主要解决了 Composition API 使用时的样板代码问题，使代码更加简洁和易读。

  • 更少的样板代码：不需要 export default 或 setup() 函数。
  • 直接使用组合式 API：可以直接使用 ref, computed 等 Vue 3 的组合式 API。
  • 模板访问：在 <script setup> 中声明的所有变量和函数都可以直接在模板中访问。
  • 类型支持：支持 TypeScript，可以直接声明 props 和自定义事件的类型
  • 性能优化：由于模板会被编译成同一作用域内的渲染函数，避免了渲染上下文代理对象，从而提高了运行时性能

KeepAlive

包裹动态组件时，会缓存不活动的组件实例，主要用于保留组件状态或避免重新渲染
作用：

  1. 缓存组件状态：当组件被包裹在 <keep-alive> 中时，即使它们从当前视图中移除，它们的状态也会被保留而不是销毁。
  2. 避免重复渲染：对于那些初始化成本较高的组件，使用 <keep-alive> 可以避免每次显示时都重新渲染，提升用户体验。
  3. 减少资源消耗：通过缓存组件实例，可以减少不必要的 DOM 操作和资源加载，节省内存和计算资源。

vueRouter传参的两种方式的区别(params和query)

params是通过路由路径传递的参数，通常用于需要在URL中明确表示的动态数据param的特点包括：

  1. 路径参数param是URL的一部分，通常以:开头，例如/user/:id。
  2. 强类型param的值在路由匹配时就被解析，适合用于需要强类型的场景。
  3. 不支持重复参数：同一路由中param不能重复。
    在用户组件中，我们可以通过this.$route.params获取到传递的参数;

query
query是通过URL的查询字符串传递的参数，通常用于可选参数或多个参数的场景。query的特点包括：

  1. 查询字符串：query是URL的一部分，格式为?key=value，可以包含多个参数，例如/user?id=1&name=John。
  2. 灵活性：query参数可以是可选的，适合用于过滤、排序等场景。
  3. 支持重复参数：同一路由中，query可以重复。
    在用户组件中，我们可以通过this.$route.query获取到传递的参数

toRef和toRefs

**toRef**函数用于将一个响应式对象(reactive)的属性转换为一个独立的ref。这个 ref 仍然与原对象保持关联，即修改这个 ref 的值会更新原对象中的相应属性，反之亦然。
**toRefs **函数用于将一个响应式对象的所有属性都转换为 refs，并返回一个包含这些 refs 的普通对象。这样可以方便地解构响应式对象，同时保持每个属性的响应性。

单向数据流

单向数据流确保了数据在组件之间的传递是单向的，即从父组件流向子组件。
单向数据流的工作原理
Vue中的单向数据流主要通过props实现。父组件通过props向子组件传递数据，子组件可以接收这些数据并使用，但不应直接修改它们。如果子组件需要修改通过props接收的数据，它应该通过事件向父组件发出信号，由父组件处理这些数据的变更。

单向数据流的优势

  1. 简化数据流动：单向数据流简化了数据在组件间的流动，使得数据的来源和变更更加清晰。
  2. 提高代码可维护性：由于数据的流动方向是单向的，当数据发生变化时，只需要关注数据的来源和受影响的组件，这使得代码更易于维护。
  3. 增强代码可预测性：单向数据流增强了对数据流动的预测性，有助于更准确地追踪和排查问题。

EventBus和Vuex

Vuex
大型项目：当你有一个大型项目并且需要管理大量状态时，Vuex 可以帮助你更好地组织和管理这些状态。
多层嵌套组件：当你的组件层级较深，需要在不同层级的组件之间共享状态时，Vuex 提供了一种清晰的解决方案。
复杂的状态逻辑：当状态变化涉及复杂的业务逻辑（如异步操作）时，Vuex 的 Action 和 Mutation 机制可以帮助你更好地组织代码。
状态调试：Vuex 提供了丰富的调试工具（如 Vue DevTools），方便你在开发过程中进行状态跟踪和调试。
EventBus
小型项目：如果你有一个小型项目并且只需要在少数几个组件之间传递数据，EventBus 是一个轻量级的选择。
局部状态管理：当只需要在少数几个组件之间共享状态，且这些状态不需要全局管理时，EventBus 是一个简便的选择。
简单通信：当需要在父子组件或兄弟组件之间进行简单的通信时，EventBus 可以快速实现这一需求。
结论
Vuex：适用于需要集中管理和维护复杂状态的应用，尤其是那些具有多层次嵌套组件和复杂状态逻辑的项目。
EventBus：适用于小型项目或局部状态管理，特别是在需要在少数几个组件之间进行简单通信时。

v-model 实现原理

v-model实际上是以下两个操作的组合：

  • v-bind（绑定属性）：将数据绑定到表单元素的值属性。
  • v-on（监听事件）：监听表单元素的变化事件，并根据事件更新数据。
    <input :value="message" @input="message = $event.target.value"/>
    在组件中
    <custom-component :model-value="message" @update:modelValue="message = $event"></custom-component>

update:modelValue 事件：这是默认的事件名称，用于通知父组件更新数据。

虚拟dom与diff算法

  • 为什么使用虚拟DOM？
    使用JS时，不可避免的会大量操作DOM，而DOM的变化又会引发回流或重绘，从而降低页面渲染性能。所以虚拟DOM出现的主要目的就是为了减少频繁操作DOM而引起回流重绘所引发的性能问题

  • 虚拟DOM是什么？作用？
    虚拟DOM是描述元素和元素之间的关系所创建的一个JS对象
    页面中的数据和状态变化时，render函数会生成一个新的虚拟DOM,DOM进行比对，找到需要修改的虚拟DOM内容，然后去对应的真实DOM中修改；不需要频繁操作，提高了页面性能；

  • 虚拟DOM和真实DOM的区别？

    特性 真实 DOM (Real DOM) 虚拟 DOM (Virtual DOM)
    本质 浏览器中的实际 DOM 树 内存中的轻量级 JavaScript 对象
    更新效率 直接更新真实 DOM，可能导致频繁重绘和回流 通过 Diffing 算法找出差异，批量更新真实 DOM
    性能影响 频繁操作真实 DOM 会导致性能问题 减少不必要的 DOM 操作，提升性能

Diff 算法的工作原理
Diff 算法的核心思想是比较两个树结构（新旧虚拟 DOM 树），找出它们之间的差异，并生成一个补丁（Patch）。这个补丁包含了需要对真实 DOM 进行的所有修改操作（如添加、删除、更新节点等）。

  2. 使用 afterEach 全局后置钩子,每次路由跳转完执行
  3. 在组件内使用 watch 监听 $route 对象
  4. 使用 beforeRouteEnter、beforeRouteUpdate 和 beforeRouteLeave 守卫(组件内的路由守卫)

vue router是如何调用生命周期的钩子函数的，

Route对象：每个路由匹配都会生成一个Route对象，包含当前路由的信息（如路径、参数等）。

SPA单页面应用与MPA多页面应用

SPA单页面应用

指只有一个主页面的应用；一开始只需要加载一次css,js等相关资源。所有的内容都包含在主页面，对每一个功能模块组件化。单页面应用跳转，就是切换相关组件，仅仅只是刷新局部资源。

MPA多页面应用

指有多个独立页面的应用，每个页面都必须重复加载js,css等相关资源。多页面应用跳转，需要整页资源刷新。

区别

  1. 刷新方式：
  • SPA:相关组件的切换，页面局部刷新或者更改
  • MPA:整页刷新
  1. 路由模式
  • SPA:可以使用hash,也可以使用history
  • MPA:普通的链接跳转
  1. 用户体验：
    • SPA：用户体验良好，切换页面只是局部刷新组件，因为是一次性加载所有资源，所以首次进入的时候可能会有一段时间的白屏。
    • MPA：页面切换加载缓慢，流畅度不够，用户体验较差
  2. 转场动画：
    • SPA:容易实现转场动画
    • MPA:无法实现
  3. 数据传递：
    • SPA:容易传递数据，方法有很多(Vuex等)
    • MPA:传递数据不方便，一般用url,cookie,本地存储
  4. 结构:
    • SPA:一个主页面+许多模块组件
    • MPA:许多完整的页面
  5. 资源文件：
    • SPA:组件共用的资源文件只需要加载一次
    • MPA:每个页面都需要加载自己的资源

new Vue与createApp

new Vue

  1. 参数
    new Vue 接收一个选项对象（Options API），包含以下常用属性：
  • el：指定挂载的 DOM 元素（如 #app）。
  • data：定义响应式数据。
  • methods：定义方法。
  • computed：定义计算属性。
  • watch：定义侦听器。
  • components：注册局部组件。
  • template：定义模板。
    全局 API：在 Vue 2 中，许多全局 API（如 Vue.component、Vue.directive）会污染全局命名空间。
    单例模式：整个应用只有一个 Vue 实例，所有功能都集中在这个实例中。
    限制：不适合复杂的应用场景，容易导致代码耦合。 createApp（Vue 3 的方式）
    (1) 使用场景
    createApp 是 Vue 3 中引入的新方法，用于创建一个独立的应用实例。
    它是 Vue 3 的推荐方式，旨在解决 Vue 2 中全局 API 污染的问题。

createApp 接收一个选项对象（Options API），与 new Vue 类似，包含以下常用属性：

  • data：定义响应式数据。
  • methods：定义方法。
  • computed：定义计算属性。
  • watch：定义侦听器。
  • components：注册局部组件。
  • template：定义模板。
