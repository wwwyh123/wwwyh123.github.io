幽灵依赖


当一个项目安装依赖A时，A又依赖于B,此时node_modules不会将A和B处理为树形结构，而是扁平化处理。也就是说在项目中可以直接引用到B。可当A被删除后，我们再访问B就会报错。

幽灵依赖特点

  • 未直接声明：这些依赖并未在你的 package.json 文件中显式列出。
  • 自动安装：当你安装一个包时，如果该包有其自身的依赖，则这些依赖会被自动安装到你的项目中。
  • 潜在问题：由于它们不是直接依赖，因此在更新或删除直接依赖时，可能会无意间影响到这些幽灵依赖，导致应用出现问题。

npm 如何解决幽灵依赖？

依旧是扁平化处理，但是你的项目只会直接依赖A

pnpm 如何解决幽灵依赖？


首先在node_modules仍是扁平化处理，但是这里的包只是一个快捷方式，指向.pnpm中的硬链接，在.pnpm中会展示依赖关系，最终.pnpm中的硬链接会指向.pnpm-store中。.pnpm-store一般存放在一个固定的位置，比如磁盘D中。
利用软链接和硬链接节省磁盘空间。
pnpm分三个阶段安装：

  1. 依赖解析。 仓库中没有的依赖都被识别并获取到仓库。
  2. 目录结构计算。 node_modules 目录结构是根据依赖计算出来的。
  3. 链接依赖项。 所有以前安装过的依赖项都会直接从存储区中获取并链接到 node_modules。

pnpm常用命令：

  • pnpm -v查看已安装的pnpm的版本
  • pnpm install xxx/pnpm i xxx安装依赖
  • pnpm run xxx运行package.json中scripts脚本
  • pnpm config get registry查看源
  • pnpm config set registry <淘宝源或私服>切换源
  • pnpm add xxx安装依赖包到 dependenciespnpm add -D xxx安装依赖包到devDependencies
  • pnpm update xxx/pnpm up xxx更新依赖包pnpm remove xxx删除依赖包
