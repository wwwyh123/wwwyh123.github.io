XSS 攻击什么是跨站脚本攻击（XSS）？常见类型有哪些（反射型、存储型等）？如何预防 XSS 攻击（如输入过滤、使用HttpOnly cookie 等）？

https://juejin.cn/post/6844903685122703367?searchId=202507141336066ACD5F0DD35AA2E2FA8A
XSS 攻击指的是跨站脚本攻击，是⼀种代码注⼊攻击。攻击者通过在⽹站注⼊恶意脚本，使之在⽤户的浏览器上运⾏，从⽽盗取⽤户的信息如cookie 等。
XSS 的本质是因为⽹站没有对恶意代码进⾏过滤，与正常的代码混合在⼀起了，浏览器没有办法分辨哪些脚本是可信的，从⽽导致了恶意代码的执⾏。

  • 攻击者可以通过这种攻击⽅式可以进⾏以下操作：
    • 获取⻚⾯的数据，如DOM、cookie、localStorage；
    • DOS攻击，发送合理请求，占⽤服务器资源，从⽽使⽤户⽆法访问服务器；
    • 破坏⻚⾯结构；
    • 流量劫持（将链接指向某⽹站）；

XSS 攻击通常注⼊在以下位置：

  1. 用户输⼊字段： 攻击者可以尝试在⽹站的⽤户输⼊字段中注⼊恶意代码，例如表单字段、搜索框、评论框等。如果⽹站没有适当地验证和过滤⽤户输⼊，攻击者可以注⼊恶意脚本，然后等待其他⽤户来访问受影响的⻚⾯。
    2.URL 参数： 攻击者可以尝试在 URL 中的查询字符串或路径参数中注⼊恶意代码。如果⽹站没有正确处理和转义这些参数，恶意代码可能会被执⾏。
  2. Cookie： 如果攻击者能够注⼊恶意脚本到⽤户的 Cookie 中，那么他们可以劫持⽤户的会话，以执⾏各种恶意操作。
  3. HTTP 头部： 攻击者可以尝试通过修改 HTTP 请求或响应的头部信息来注⼊恶意代码。这可能导致其他⽤户在访问⽹站时受到攻击。
  4. 内联脚本： 如果⽹站允许⽤户在⻚⾯中插⼊内联脚本，攻击者可以尝试在这些内联脚本中注⼊恶意代码。这是⼀种很危险的情况，因为注⼊的脚本将被直接执⾏。
  5. 第三⽅内容： 如果⽹站允许第三⽅内容（例如⼴告、社交媒体⼩部件等）嵌⼊到⻚⾯中，攻击者可以尝试在这些第三⽅内容中注⼊恶意代码，以影响⻚⾯的访问者。
    如何防御
  • 对⽤户输⼊进⾏过滤和转义，以防⽌恶意脚本被注⼊。
  • 使⽤ CSP（ 内容安全策略 ） ，CSP 的本质是建⽴⼀个⽩名单，告诉浏览器哪些外部资源可以加载和执⾏，从⽽防⽌恶意代码的注⼊攻击。
  • 使⽤ HTTPOnly 标记的 Cookie，以防⽌ Cookie 被窃取。在⽣成cookie时使⽤HttpOnly标志有助于减轻客户端脚本访问受保护cookie的⻛险

XSS攻击通常分为以下几种主要类型：

  1. 反射型XSS（Non-Persistent XSS）
    描述：反射型XSS是最常见的XSS攻击形式之一。在这种攻击中，恶意脚本通过URL参数或表单输入等方式“反射”到受害者的浏览器上。例如，攻击者可能会发送一个包含恶意脚本的链接给受害者，当受害者点击该链接并访问目标网站时，恶意脚本将被执行。
    例子：假设有一个搜索功能，它直接将用户的搜索查询返回到结果页面中。如果攻击者构造了一个带有<script>标签的搜索请求，并诱导他人点击这个链接，则接收者的浏览器将会执行这段脚本。
  2. 存储型XSS（Persistent XSS）
    描述：存储型XSS涉及将恶意脚本存储于服务器端数据库或其他持久化存储介质中。一旦被存储，每当有用户访问含有该恶意脚本的数据时，脚本就会自动执行。这使得攻击更加危险，因为它不需要每次都欺骗用户去点击特定链接。
    例子：在一个社交网络应用中，如果评论系统没有正确过滤用户提交的内容，攻击者可以发布一条包含恶意脚本的评论。之后任何查看这条评论的人都可能成为攻击的目标。
  3. 基于DOM的XSS
    描述：不同于前两种类型依赖于服务器端处理过程，基于DOM的XSS完全发生在客户端。这类攻击利用了网页文档对象模型（DOM）结构中的某些特性，使得恶意输入能够改变页面逻辑而不经过服务器验证。
    例子：考虑一个使用JavaScript动态更新页面内容的应用程序，其中部分HTML内容是根据URL片段标识符（#后面的部分）生成的。若这部分未经过适当编码就插入到了DOM中，则攻击者可以通过构造特殊格式的URL来触发XSS。
对比维度 反射型XSS DOM型XSS
恶意代码处理位置 服务器端处理（服务器将恶意代码插入HTML）。 客户端处理（前端JavaScript直接操作DOM）。
是否需要服务器响应 需要：恶意代码必须由服务器返回并插入到页面中。 不需要：恶意代码由前端JavaScript动态生成，服务器无需参与。
典型攻击场景 搜索框、跳转链接、错误页面等依赖服务器返回内容的场景。 URL片段（hash）、前端路由（如 history.pushState）、客户端动态渲染的场景。
防御难度 相对简单：可通过过滤服务器端输出的恶意代码（如HTML转义）防御。 较复杂：需严格限制前端JavaScript对DOM的修改（如避免使用 innerHTML）。

CSRF

https://tech.meituan.com/2018/10/11/fe-security-csrf.html
CSRF 攻击指的是跨站请求伪造攻击，攻击者诱导⽤户进⼊⼀个第三⽅⽹站，然后该⽹站向被攻击⽹站发送跨站请求。如果⽤户在被攻击⽹站中保存了登录状态，那么攻击者就可以利⽤这个登录状态，绕过后台的⽤户验证，冒充⽤户向服务器执⾏⼀些操作。
攻击者通常会在其他⽹站上设置⼀个包含恶意请求的链接或表单，当受害者点击此链接或提交表单时，浏览器会⾃动发送请求到⽬标⽹站，由于受害者已经登录了⽬标⽹站，所以请求会被认为是合法的。
大白话就是：用户登录了目标网站之后访问了第三方恶意网站，恶意网站有向目标网站做某些操作的代码，会被浏览器执行，由于此时浏览器还有cookie，所以可以向目标网站发送请求。
CSRF 攻击的本质是利⽤ cookie 会在同源请求中携带发送给服务器的特点，以此来实现⽤户的冒充。

  • 进⾏同源检测 使⽤请求中的Referer字段。Referer字段包含了请求的来源URL。如果发现请求的Referer字段与服务器期望的不同，则可以拒绝请求。
  • 在请求地址中添加token并验证
    • 在 HTTP 请求中以参数的形式加⼊⼀个随机产⽣的 token，并在服务器端建⽴⼀个拦截器来验证这个 token，如果请求中没有 token 或者 token 内容不正确，则认为可能是 CSRF 攻击⽽拒绝该请求。
  • 使⽤SameSite属性的Cookie。在Cookie中设置SameSite属性可以限制Cookie的发送范围。
  • 使⽤ HTTPS 加密通信，以防⽌信息被窃取。

HttpOnly 值为 true 或 false,若设置为true，则不允许通过脚本document.cookie去更改这个值，同样这个值在document.cookie中也不可⻅，但在发送请求时依旧会携带此Cookie。

samesite ⽤来限制第三⽅ cookie 在Cookie中设置SameSite属性可以限制Cookie的发送范围以此来防⽌ CSRF 攻击，有三个属性 Strict（任何情况都不发送）、Lax（⼤多数情况也是不发送第三⽅Cookie，但是导航到⽬标⽹址的 Get 请求除外。）、None（关闭）
